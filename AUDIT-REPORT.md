# Аудит hours-calculator — 2026-07-05

> Независимый аудит перед боевым запуском (заходят реальные работники, реальные деньги).
> **Режим: только отчёт.** Код приложения не менялся, ничего не деплоилось. Единственный
> новый файл — этот отчёт. Проверки на живом API — только read-only (GET, unauth-пробы, CORS).
> Аудит вёлся в 4 независимых потока: доступ/права (вручную + живые пробы), корректность
> расчётов, приватность/качество кода, стабильность/edge-cases.

## Резюме

- **Проверено эндпоинтов:** ~30 (auth, workers, shifts, team, payouts, summary, receipts, settings, push, health).
- **Найдено:** **3 критичных 🔴**, **12 важных 🟡**, **9 желательных 🟢** + список UX-идей.
- **Общая оценка готовности к боевому запуску:** **ЕСТЬ БЛОКЕРЫ (немного).**
  Фундамент прочный — авторизация, изоляция ролей, снапшот ставок, приватность отчёта боссу,
  отсутствие секретов в коде — всё сделано правильно и подтверждено. Но перед заходом
  работников нужно закрыть 3 🔴 (дубли смен, таймаут Gemini, валидация времени) — все три
  малозатратные. После них — боевой запуск возможен, остальное чинится итеративно.

**Что радует:** ни одной дыры «работник видит чужие деньги/данные», ни одного пути «работник
подделывает роль», ни одного секрета в git. Слабые места — про **надёжность и валидацию входа**,
а не про фундаментальную безопасность.

## 🔴 КРИТИЧНО (чинить до захода работников)

### 🔴-1. Двойной клик «Сохранить» создаёт дубль смены → задвоение зарплаты
- **Что:** на таблице `shifts` нет ни одного UNIQUE-ограничения (только PK `id`). Два одинаковых POST создают две строки, обе попадают в `SUM(hours × rate)`.
- **Где:** `api/migrations/001_init.sql` (таблица `shifts` — нет UNIQUE); `api/main.py:277-286` (INSERT без ON CONFLICT); суммирование `api/logic.py:63-73`.
- **Риск:** работник на мобильном/PWA при медленной сети тапает «Сохранить» дважды → смена посчитана дважды → завышенный `earned_by_hours` → ложная недоплата/спор. **Не требует злого умысла — происходит само** на плохой сети.
- **Как воспроизвести:** дважды отправить `POST /shifts` с одинаковым телом (worker_id, date, object_name, start/end). Оба вернут 200 с разными id; `GET /shifts` покажет две; заработок недели удвоится.
- **Рекомендация:** partial UNIQUE (`worker_id, date, start_time, end_time, object_name`) + `INSERT ... ON CONFLICT DO NOTHING`, либо серверный idempotency-key. Блокировки кнопки на фронте недостаточно.

### 🔴-2. У вызова Gemini нет таймаута → загрузка чека может зависнуть намертво
- **Что:** `generate_content(...)` вызывается без `request_options={"timeout": ...}`. У `google-generativeai` нет короткого дефолтного таймаута, а вызов идёт внутри `asyncio.to_thread` — зависшее соединение держит поток пула, и запрос `/receipts/upload` не завершается никогда. `try/except` ловит исключения, но НЕ зависание.
- **Где:** `api/gemini.py:84-87` (нет таймаута), `api/gemini.py:109` (`to_thread`), вызов инлайн до записи в БД — `api/receipts.py:86`.
- **Риск:** при тормозах/сетевом сбое Gemini загрузки виснут; дефолтный пул потоков (~40) исчерпывается → встаёт всё приложение, вместо чистой ошибки. Учитывая уже висящую **квоту 429**, это реальный сценарий.
- **Рекомендация:** `request_options={"timeout": 15}` в `generate_content` и/или `asyncio.wait_for(...)`. При таймауте — деградация в `pending_review` (этот путь уже есть).

### 🔴-3. `start_min` / `end_min` без границ → работник может внести ~999 часов за смену
- **Что:** ничего не проверяет, что время в `[0,1439]`. `preview_shift` считает `gross = end − start` и заворачивает только при `gross < 0`. Работник может отправить `start_min=0, end_min=59940` → ~999 часов в одной смене. При этом сохранённые `start_time/end_time` заворачиваются по модулю (`% 1440`) и показывают нормальное время — карточка выглядит обычной, а `calculated_hours` огромный.
- **Где:** модели `api/main.py:187-197` (`int` без ограничений); `_min_to_time` `api/main.py:38-41`; `calc.preview_shift` `api/calc.py:31-34`.
- **Риск:** завышенные часы напрямую кормят `earned_for_week` → ложная «недоплата» на тысячи долларов, спор с боссом. Через обычный UX (пикеры времени) не случится — нужен ручной запрос с валидным токеном, но у работника токен есть.
- **Частичная защита:** `calculated_hours numeric(5,2)` ограничен 999.99 → значения ≥1000ч дают Postgres-overflow (500, некрасиво, но не сохраняется молча). Всё, что <1000ч, сохраняется штатно.
- **Смягчающий фактор бизнес-модели:** боссу уходят только часы, а сумму выплаты он вводит сам по чеку — прямой кражи денег нет, но искажается запись «сколько должны».
- **Рекомендация:** `Field(ge=0, le=1439)` на `start_min/end_min` в обеих моделях; ограничить gross ≤ 24ч; заодно уйдёт и 500 из численного переполнения.

## 🟡 ВАЖНО (в ближайшее время)

### 🟡-1. Один чек может обеспечить выплаты за ДВЕ разные недели (нет UNIQUE на receipt_id)
- **Где:** проверка «чек уже использован» `api/payouts.py:180` вне транзакции; `weekly_payouts` UNIQUE только по `(worker_id, week_start)` (`001_init.sql`), не по `receipt_id`.
- **Риск:** два конкурентных `from-receipt` с одним чеком, но разными `week_start`, оба проходят проверку и вставляются. Та же неделя надёжно блокируется UNIQUE→409, это более узкий край.
- **Рекомендация:** `UNIQUE (receipt_id)` (partial where not null) на `weekly_payouts`, либо проверка внутри транзакции с `FOR UPDATE`.

### 🟡-2. Размер файла проверяется ПОСЛЕ чтения всего тела → DoS диском/памятью
- **Где:** `api/receipts.py:63-75` — `data = await file.read()` грузит весь upload до проверки `MAX_SIZE` (5 МБ); `content_type` client-supplied, подделывается на `image/jpeg`.
- **Риск:** клиент шлёт многогигабайтное тело с `Content-Type: image/jpeg` — проходит MIME-проверку, всё тело буферизуется (Starlette спулит на диск после ~1 МБ) до отклонения. Дефолтного лимита тела в uvicorn нет.
- **Рекомендация:** лимит тела на уровне ASGI/прокси, либо чтение чанками с прерыванием при превышении.

### 🟡-3. Планировщик дублирует пуши, если процессов/реплик больше одного
- **Где:** `api/scheduler.py` — APScheduler в памяти процесса, без межпроцессной блокировки; стартует в каждом процессе (`main.py:62`). Advisory-lock миграций планировщик НЕ покрывает.
- **Риск:** при `--workers >1` или ≥2 репликах Railway каждый напоминалка-пуш уходит N×. Сегодня 1 реплика — ок, но ничего это не гарантирует.
- **Рекомендация:** leader-election через Postgres advisory-lock, либо отдельный single-instance воркер. До этого — зафиксировать в деплое: API строго 1 реплика / 1 worker.

### 🟡-4. Нет rate-limit на `/auth/login` → возможен перебор паролей
- **Где:** `api/auth.py:42-56` — нет троттлинга/локаута. (Тайминг-атака закрыта dummy-хэшем — это плюс.)
- **Риск:** брутфорс пароля работника без ограничений.
- **Рекомендация:** троттлинг по IP/email (напр. slowapi), либо экспоненциальный локаут.

### 🟡-5. Нет глобального обработчика ошибок → сырые 500 при сбоях БД
- **Где:** обёртки `api/db.py:29-50` без try/except; нет `@app.exception_handler`; пул `max_size=5` (`db.py:24`).
- **Риск:** при кратком сбое БД / насыщении пула (Gemini/push держат запросы) пользователь видит голый 500 без ретрая и понятного текста.
- **Рекомендация:** глобальный JSON-обработчик ошибок; небольшой ретрай на транзиентных asyncpg-ошибках; пул побольше.

### 🟡-6. Неделя Пн–Вс на стыке месяцев показывается в ДВУХ месяцах (задвоение в месячных итогах)
- **Где:** `frontend/src/lib/weeks.ts:20-32` (`getWeeksInMonth`) — `mondayOf(1 июля)=29 июня`, поэтому пограничная неделя попадает и в июнь, и в июль. Деньги в БД не задваиваются (UNIQUE не даст создать вторую выплату), но месячный роллап, итерирующий `getWeeksInMonth`, посчитает неделю дважды.
- **Риск:** путаница в сверке: показанные месячные часы ≠ заработок недели выплаты.
- **Рекомендация:** привязать каждую неделю Пн–Вс к одному месяцу (напр. по четвергу недели или по `week_start`); в месячных итогах дедуп по `week_start`.

### 🟡-7. Работник может сам поднять себе ставку (модель Слоя 7f) — подтвердить, что так задумано
- **Где:** `api/auth.py` PATCH `/auth/me` — снят supervisor-гейт на `hourly_rate`; worker меняет свою ставку (снапшот прошлых смен не пересчитывается, идёт push супервайзеру).
- **Риск:** работник в одностороннем порядке завышает будущий `earned_by_hours`; единственная защита — пуш супервайзеру + ревью. Это осознанное решение 7f — **фиксирую как момент для подтверждения**, не как дефект.
- **Рекомендация:** если нежелательно — вернуть подтверждение супервайзером; иначе оставить как есть (push уже есть).

### 🟡-8. Дневная напоминалка «Внеси смену» без маркера идемпотентности — до 4×/день
- **Где:** `api/scheduler.py:45-53` шлёт в 19/20/21/22 локального, если смены за день нет, без записи «уже напоминал». (Субботняя недельная — правильно дедупится через `weekly_reports`.)
- **Риск:** не краш и не дубль от рестарта (jobstore в памяти), но шумно — до 4 напоминаний/день работнику без смены.
- **Рекомендация:** если не задумано — маркер «отправлено за день» или один час напоминания. Подтвердить намеренность.

### 🟡-9. `elif body.hours` в правке смены сохраняет часы БЕЗ клампа (латентно)
- **Где:** `api/main.py:439-441` — если у смены `start_time` ИЛИ `end_time` = NULL, ветка пишет `body.hours` как есть, без ограничения `{hours_down, hours_up}`.
- **Риск:** работник правкой такой смены поставил бы произвольные часы (≤999.99). **Сейчас недостижимо** — `create_shift` всегда пишет оба времени, backfill тоже; NULL-времена бывают только у ручных/легаси-строк.
- **Рекомендация:** в `elif` всё равно кламп/reject, либо убрать ветку сырых часов.

### 🟡-10. Мёртвые `calc.RATE` / `calc.money` — «заряженная» неверная выплата (БЫСТРЫЙ ФИКС)
- **Где:** `api/calc.py:9` `RATE = 25` и `api/calc.py:41-42` `money(hours) = round(hours*RATE, 2)`.
- **Риск:** функция не используется (живой код зовёт `logic.money(hours, snapshot)`), но хардкодит плоские $25/ч мимо снапшота. Любой будущий вызов молча заплатит всем $25/ч. Не эксплуатируется сейчас, но это футган.
- **Рекомендация:** удалить `RATE` и `calc.money`; оставить `LUNCH_MIN`/`preview_shift`.

### 🟡-11. Хардкод, который место в конфиге
- **Где:** `api/gemini.py:27` `_MODEL_NAME="gemini-2.0-flash"` (модель хардкодом — при квоте 429 нельзя сменить без редеплоя); `api/db.py:118` сид supervisor `hourly_rate=27.00` в SQL; `api/config.py:23` `VAPID_SUBJECT` дефолт — личный mailto; `.env.example` содержит реальный `OWNER_ID=7180071033`.
- **Рекомендация:** вынести модель Gemini и стартовую ставку в env; убрать реальный OWNER_ID из примера.

### 🟡-12. Легаси-связка `OWNER_ID` как ключ арендатора — починить ДО второго супервайзера
- **Где:** `api/deps.py:70` — `CurrentUser.id = OWNER_ID` для КАЖДОГО пользователя; фильтрация строк идёт по `user_id = OWNER_ID`.
- **Риск:** сейчас один супервайзер — всё ок. Но если засеять второго супервайзера/команду, все схлопнутся в одного арендатора → второй увидит смены, выплаты и деньги первой команды. Комментарий в `deps.py:8-13` это признаёт.
- **Рекомендация:** до появления второй команды — брать арендатора из строки пользователя (колонка-владелец), а не из глобальной константы `OWNER_ID`.

## ✅ ЧТО ХОРОШО (не трогать)

**Авторизация и роли:**
- JWT `role`/`worker_id` в токене декоративны — `require_auth` перечитывает их из БД, подделка роли в токене бесполезна (`deps.py:57-77`). Подпись проверяется, алгоритм зафиксирован `HS256`, секрет обязателен ≥32 симв. (fail-fast на старте).
- **Все защищённые эндпоинты отдают 401 без токена** — проверено живьём: /shifts, /team, /payouts, /settings, /summary, /workers, /auth/me.
- **CORS в проде заужен на домен Vercel** — проверено живьём (evil-origin без ACAO).
- Изоляция работника enforced на КАЖДОМ read-пути по `worker_id`, client-supplied `worker_id` для worker игнорируется: shifts GET/PATCH/DELETE (`_shift_for_edit`), payouts, receipts (`_accessible_receipt`), summary (`_resolve_worker`), workers.
- Supervisor-гейты на месте: `/team*`, `PATCH /settings`, `PATCH /receipts/{id}/review`, `/workers` create/patch, `rate_override`, `suppress_notification`.
- bcrypt + константное время логина (dummy-хэш против перечисления), нет открытой регистрации, `password_hash` нигде не возвращается, отзыв сессий через `token_version`.
- Весь SQL параметризован (`$1..$n`), включая динамические `SET` (имена колонок — литералы, значения — параметры). Инъекций не найдено.

**Расчёты (деньги):**
- Округление часов (`<15` вниз / `>20` вверх / `15–20` выбор) — границы верны; путь выбора НЕ эксплуатируется (`_resolve_hours` принимает клиентские часы, только если совпадают с одним из двух валидных значений).
- Смена через полночь, обед (−30 однократно), нулевая длительность (→0, без краха) — корректны и в create, и в update.
- **Деньги = часы × `hourly_rate_snapshot`**, снапшот фиксируется при создании и НЕ пересчитывается при правке ставки/смены. Смена ставки применяется только к будущим сменам (Вариант A).
- Бонус/недоплата (`max(0, paid−earned)` / `max(0, earned−paid)`) — знаки верны; `fine` требует заметку. Недели Пн–Вс, воскресенье в правильной неделе, UNIQUE `(worker_id, week_start)` не даёт задвоить выплату.
- Таймзона в платёжном пути чистая: даты явные `YYYY-MM-DD`, без серверных TZ-сдвигов; «сегодня»/напоминания — `America/Los_Angeles`.

**Приватность:**
- **Отчёт боссу (SMS) — только часы, денег нет по построению** (`formatWeekReport`/`ReportShift` не содержат ставку/сумму даже в типе).
- Пуши не текут между работниками; денежные пуши идут только супервайзеру.
- Ответы API не переэкспонированы (нет хэшей/внутренних полей).

**Надёжность (частично):**
- Gemini 429/парс-ошибки деградируют мягко: фото сохраняется, `pending_review`, upload успешен. Не-картинки отсекаются PIL (ловит подделку content-type на реальном декоде).
- Мёртвые push-подписки чистятся на 410/404 (`SubscriptionExpired`). Мультиустройство — чистый цикл (3 устройства = 3 отправки, без рекурсии).
- Субботняя напоминалка идемпотентна через `weekly_reports`.
- Все миграции идемпотентны (`IF NOT EXISTS`, `ON CONFLICT DO NOTHING`), в правильном порядке под advisory-lock с релизом в `finally`. Повторный прогон безопасен.
- Секретов в коде и истории git НЕТ (`git log -p --all` чист); `.gitignore` корректен; секреты только из env.

## 💡 UX / улучшения (Блок 6 — идеи, не баги)

- **Блокировка кнопки «Сохранить» + оптимистичный спиннер** на время POST — частично прикроет 🔴-1 на фронте (в дополнение к серверному UNIQUE). Польза высокая, сложность низкая.

## 1. Карта системы

- **Бот** `dist/workspace-telegram-bot.js` (long-poll, tmux-инъекция), **режим `INJECTION_MODE=tmux`**,
  tmux 3.7b. Claude запускается как **tmux-сессия** (независима от процесса бота → переживает его рестарт).
- **Хуки** (отдельные процессы, stdin-payload, шлют в Telegram): `enhanced-hook-notify.js` (завершение/
  ожидание/…), `permission-hook.js` (кнопки Allow/Deny, блокирующий опрос), `question-notify.js` и др.
- **`workspace-router.js`** — карта сессий (файл `src/data/session-map.json`), история проектов, resolveWorkspace.
- Связь бот↔хуки — файлами (`/tmp/claude-prompts/`, `src/data/typing-active`, карта сессий).
- **launchd `com.ccgram`** — keepalive + runatload.
- `smart-monitor.js` присутствует, но **не используется** (мёртвый компонент).

## 4. Сессии / tmux / доставка — ГЛАВНОЕ

- **Переживание рестарта бота — ТЕПЕРЬ РАБОТАЕТ (было сломано).** Живой тест: 2 tmux-сессии →
  `kickstart` бота → tmux-сессии **живы**, НО карта сессий очищалась (истёкшие записи), и бот их
  «терял» → пришлось бы `/new`. **Починил:** на старте в tmux-режиме бот синхронизирует карту с
  `tmux ls` — **усыновляет живые сессии**, убирает мёртвые. Лог: `adopted 2 live tmux session(s)`.
  После этого `resolveWorkspace('hours-calculator')` → exact, `tmux has-session` → жива. Задачу
  можно слать **без нового `/new`**. (Финальный end-to-end — твой тест из Telegram.)
- **Session not found** (старый баг pty) — в tmux-режиме не воспроизводится: источник истины —
  tmux-сервер, бот сверяется с ним.
- **Переживание перезагрузки Mac — ЧЕСТНО: НЕТ.** tmux-сервер умирает при reboot Mac → сессии
  пропадают. Переживают только рестарт/краш самого бота, не reboot. (Для reboot нужен tmux
  `@resurrect`/`@continuum` — отдельная настройка, если надо — скажи.)
- **SESSION_TIMEOUT 24ч → 720ч** (в `.env`) — чтобы живая tmux-сессия не «истекала» в карте за
  сутки простоя без рестарта.
- **Уведомление о завершении** — реальный прогон хука: дошло (exit 0, лог ошибок пуст).
- **Кнопки разрешения** — доставляются (создался `pending-*.json` → кнопка ушла); тап-резолв
  верен по коду.
- **Длинные сообщения** — chunking (юнит: 21000 → 6 кусков ≤4096).
- **Долгая задача** — маркер typing-active до 3ч + `alwaysNotify` → уведомления/разрешения доходят.

## 5. Устойчивость к зависаниям (наша боль — 2ч на brew)

- **Таймаута на задачи НЕТ** (был). Активного детектора зависаний НЕТ (`smart-monitor.js` спит).
- `/stop` **реально прерывает** (C-c) — живой тест прошёл. Для зависшего Claude C-c прерывает его текущий шаг.
- **ДОБАВИЛ: предупреждение о долгой задаче.** Если задача из Telegram идёт > **20 мин** и не
  завершилась — ОДНО уведомление «⏳ идёт 20 мин, /status посмотреть, /stop прервать». Alert-only,
  ничего не убивает. Это же ловит и случай «промпт вставлен, но не отправлен» (маркер жив, завершения нет).
- **sudo/пароль:** Claude в tmux-pane без TTY для пароля → sudo зависнет. Твоё правило «не ставить
  с паролем» правильное; **мост НИКОГДА не должен запускать привилегированные/интерактивные команды**.
- **ПРЕДЛАГАЮ (не делал — нужна живая настройка):** настоящий детектор зависаний — раз в N минут
  снимать `tmux capture-pane`; если в pane есть индикатор работы (`esc to interrupt`/спиннер ✻✳✶),
  но контент (без анимации спиннера) не меняется > M минут → «возможно зависла». Точную настройку
  порогов и отсев ложных срабатываний надо обкатать вживую — скажи, включу.

## 6. Безопасность (важно перед мостом)

- **ID-lock ✅:** `chat.id !== CHAT_ID` в обработчиках сообщений и callback — чужой чат игнорируется.
- ⚠️ `TELEGRAM_WHITELIST` в коде **не используется** (замок только на `CHAT_ID`; практически то же).
- **Секреты ✅:** `~/.ccgram` не git-репо, токена в репозиториях нет, в логах не светится (редактирую при показе).
- **Инъекции — НАЙДЕНА И ЗАКРЫТА ДЫРА:**
  - **Текст задачи — безопасен** (реальный тест: `'; touch /tmp/PWNED; '` не пробил — shell-экранирование держит).
  - **Имя воркспейса — БЫЛО уязвимо:** санитайзация была слабой (`[.:\s]→-`), метасимволы `; $ ` \``
    проходили → **command injection через имя проекта/воркспейса** (критично для моста!). **Починил:**
    строгая санитайзация `[^a-zA-Z0-9_-]→-` во всех 4 точках + кавычки на send-keys. Реальный ре-тест:
    `proj;whoami;x` → `proj-whoami-x`, инъекция не проходит.
- **Для МОСТА (обязательно):** (1) у моста должен быть СВОЙ auth (Telegram-замок его не покрывает);
  (2) любой параметр воркспейса/проекта из моста санитайзить так же строго; (3) не запускать
  привилегированные команды; (4) в идеале перейти на `execFile`/`spawn` с массивом аргументов (без shell).

## 7. Стабильность 24/7

- **Автоперезапуск после краша ✅** (в заходе-1 живой тест: `kill -9` → launchd поднял за ~2с). keepalive.
- **Reboot Mac:** сам бот поднимется (runatload), но **tmux-сессии не переживут** (см. п.4).
- **Логи в файл ✅:** `bot-stdout/stderr.log` + `hook-notify.log` (видимые ошибки отправки из хука).
- Утечки: pty-буфер обрезается (в tmux-режиме не задействован); индикатор «печатает» до 3ч — много API-вызовов (минор).

## 8. Список проблем по приоритету

- 🔴 **Инъекция через имя воркспейса** — **ПОЧИНЕНО** (критично для моста).
- 🔴 **Сессии терялись после рестарта бота** — **ПОЧИНЕНО** (усыновление из `tmux ls`).
- 🟡 **Reboot Mac убивает tmux-сессии** — не чинил (нужен tmux-resurrect); честно задокументировано.
- 🟡 **Нет настоящего детектора зависаний** — добавил 20-мин предупреждение; полный детектор предложен.
- 🟡 **Несколько unquoted `${name}` в exec** (напр. sessionInterrupt) — безопасны после строгой
  санитайзации у источника, но кавычки проставлены не везде (defense-in-depth неполный).
- 🟡 `TELEGRAM_WHITELIST` не enforced (замок chat.id есть).
- 🟢 `smart-monitor.js` спит; правки в `dist` затрутся при обновлении CCGram.

## 12. Итог: готов ли как фундамент для HTTP-моста?

**Почти — после этого захода критичное закрыто.** Инъекция через имя воркспейса (главная дыра для
моста) устранена и проверена; сессии переживают рестарт бота. **Прежде чем строить мост, обязательно:**
(1) свой auth у моста, (2) санитайзация всех параметров от моста, (3) запрет привилегированных команд,
(4) желательно execFile вместо shell. С этими условиями фундамент — надёжный. Открытый честный минус:
tmux-сессии не переживают перезагрузку Mac (только рестарт бота).

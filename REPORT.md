# Отчёт: оптимизация N+1 запросов в выплатах (аудит 🟢)

Дата: 2026-07-06. Файл: `api/payouts.py`.

## 1. Проблема

В листинге выплат (`GET /payouts`) для **каждой** выплаты делались отдельные запросы:
- `_covered_shifts()` — смены, покрытые выплатой (JOIN payout_shifts+shifts).
- запрос к `receipts` (review_status/review_note), если у выплаты есть чек.

Итого на N выплат — до `1 + 2N` запросов к БД вместо `1 + 2`. С ростом числа выплат
список тормозил бы линейно.

## 2. Что сделано

- Разделил `_enrich()` на:
  - `_enrich()` — как раньше, для одиночной выплаты (используется в create/patch/delete,
    там и был/остаётся 1 payout — N+1 там не проблема).
  - `_build_enriched()` — чистая сборка результата из уже загруженных данных, без
    собственных запросов (кроме fallback `earned_for_week` для очень старых выплат без
    `payout_shifts` — редкий случай, оставлен как есть).
- В `list_payouts()`:
  - все `covered_shifts` для всех выплат списка грузятся **одним** запросом
    (`WHERE ps.payout_id = ANY($1::uuid[])`), затем раскладываются по payout_id в памяти.
  - все чеки (`receipts`) грузятся **одним** запросом (`WHERE id = ANY($1::uuid[])`).
- Бизнес-логика (earned/bonus/shortfall, снапшот ставки) не менялась.

## 3. Проверка (реальный тест, не «должно работать»)

Тестов в проекте нет (нет тестовой БД), поэтому проверял напрямую на реальной Railway
Postgres через `railway variables` (только `SELECT`/временная транзакция с `ROLLBACK`,
данные не изменены):

- **Прогнал сам эндпоинт** `payouts.list_payouts()` против прод-БД (supervisor,
  1 существующая выплата) — отработал без ошибок, вернул корректные данные.
- **Сравнил старый и новый подход на 5 синтетических выплатах** (созданы во временной
  транзакции, откачена в конце, в БД ничего не осталось):
  - **Старый подход:** 5 выплат → **5 запросов** на covered_shifts (по одному на выплату).
  - **Новый подход:** 5 выплат → **1 запрос** на covered_shifts (batch).
  - **Данные идентичны** (`match: True`) — те же id смен на каждую выплату, в том же порядке.

## 4. Файлы / коммит

- `api/payouts.py` — batch-загрузка covered_shifts и receipts в `list_payouts`.
- Коммит: `perf: batch-load payouts to fix N+1 (audit 🟢)`.

## 5. Деплой

- Бэкенд запушен в `main` → задеплоится на Railway автоматически.
- Фронтенд не трогался.

## 6. Риски / рекомендации

- Fallback `earned_for_week` для legacy-выплат (без `payout_shifts`) остаётся построчным
  — это редкий случай (старые записи до введения покрытия смен), не оптимизировал
  намеренно, чтобы не трогать отдельную бизнес-логику расчёта по неделе.
- Рекомендую после деплоя проверить `/payouts` в проде с реальным числом записей
  (сейчас в проде всего 1 выплата — не показательно по нагрузке, но корректность данных
  подтверждена на синтетических данных).

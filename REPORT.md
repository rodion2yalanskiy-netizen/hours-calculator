## 1. Карта системы

- **Основной бот** `~/.ccgram/dist/workspace-telegram-bot.js` — long-polling, обрабатывает
  команды, кнопки (callback), инъекцию текста в Claude.
- **Режим ввода:** `INJECTION_MODE=pty` (tmux НЕ установлен). Claude запускается как
  **дочерний pty-процесс бота** (`src/utils/pty-session-manager.js`), хендлы — **в памяти процесса**.
- **Хуки** (Claude Code запускает их по событиям; каждый — отдельный процесс, читает payload
  из stdin, шлёт в Telegram напрямую; прописаны в `~/.claude/settings.json`):
  - `enhanced-hook-notify.js` — Stop/Notification (завершение, ожидание, смена каталога…)
  - `permission-hook.js` — запрос разрешения (кнопки Allow/Deny, блокирующий опрос ответа)
  - `question-notify.js`, `elicitation-notify.js`, `user-prompt-hook.js` и др.
- **`workspace-router.js`** — карта сессий (файл `src/data/session-map.json`), история проектов,
  `resolveWorkspace`.
- **Связь бот↔хуки — через файлы:** `/tmp/claude-prompts/pending-*.json` / `response-*.json`
  (разрешения), `src/data/typing-active` (флаг «задача из Telegram»), карта сессий.
- **launchd** `com.ccgram` — `keepalive` + `runatload` (24/7).

## 4. Сессии и доставка (главные баги)

- **«Session not found» — КОРНЕВАЯ ПРИЧИНА найдена и починена.** В pty-режиме сессии Claude
  живут в памяти процесса бота, но карта сессий — в файле. При **любом рестарте бота** (краш,
  автоперезапуск, перезагрузка Mac, мой kickstart) pty-процессы умирают, а файл всё ещё
  показывает их «активными» → фантомы → `/sessions` и карточка показывают мёртвую сессию,
  инъекция падает «Session not found». **Фикс:** на старте бот чистит осиротевшие сессии.
  **Тест:** после рестарта в логе `Reconciled session map: cleared 4 orphaned session(s)`,
  карта стала пустой.
- **Уведомление о завершении** — работает (прошлый фикс `alwaysNotify`). **Тест реальной
  отправкой:** прогнал хук `completed` — сообщение дошло в Telegram (exit 0, лог ошибок пуст).
- **Кнопки разрешения** — **доходят.** Прогнал `permission-hook.js`: создался `pending-*.json`
  (значит хук дошёл до отправки кнопки), + контрольная inline-кнопка доставлена (`ok:true`).
  Обработка тапа (`perm:` → запись ответа → хук читает) верна по коду; финальный тап — за тобой.
- **Длинные сообщения** — разбиваются. Юнит-тест: 21000 символов → 6 кусков ≤4096. `/status`
  теперь тоже chunked на обоих путях.
- **Долгие задачи (>5/>30 мин)** — маркер `typing-active` продлён до 3ч (прошлый фикс) +
  `alwaysNotify` → уведомления и разрешения доходят.

## 5. Edge cases

Проверил (бот не падал ни на одном):
- Пустой текст / пробелы → `if (!text) return` (игнор).
- Мусорные аргументы (`/use несуществующий`) → `resolveWorkspace` вернул `{type:'none'}` → адекватная ошибка.
- Устаревший callback старой карточки → `handleTaskCardCallback` проверяет `_taskCard` → «Карточка устарела».
- Кириллица/юникод → работает.
- Ломающий Markdown (`* _ [ ]`) → `sendMessage` при ошибке парсинга шлёт простым текстом (фикс есть).
- Быстрая серия сообщений → обрабатываются последовательно в цикле опроса (гонки не видно; инъекция
  в pty возвращается быстро, не блокирует цикл). Отдельной очереди нет — новая задача просто
  до-инъектится в ту же сессию (Claude сам обрабатывает по очереди).

## 6. Безопасность

- **ID-lock ✅:** и `processMessage`, и `processCallbackQuery` проверяют `chat.id !== CHAT_ID`
  → чужой чат игнорируется. В приватном чате `chat.id` = твой id, обхода нет.
- ⚠️ **`TELEGRAM_WHITELIST` в коде НЕ используется** (0 ссылок) — замок держится только на
  `CHAT_ID`. Практически это тот же результат (бот отвечает только твоему чату), но переменная
  декоративна. Рекомендация: либо убрать из `.env`, либо (лучше) добавить проверку `from.id` по whitelist.
- **Секреты ✅:** `~/.ccgram` — НЕ git-репозиторий (токен закоммитить нельзя), в репо
  hours-calculator токена нет. Токен только в `~/.ccgram/.env`.
- **Инъекции ✅:** в pty-режиме бот пишет **сырой текст в Claude REPL**, не в shell — твой
  промпт не может стать shell-командой через бота. (То, что делает уже сам Claude с
  --skip-permissions — отдельный принятый тобой риск.)
- 🟢 Одна shell-интерполяция (`tmux capture-pane -t ${name}`) с именем проекта — **не
  достигается в pty-режиме**, имя = имя git-папки (не произвольный ввод). Низкий риск.

## 7. Стабильность 24/7

- **Автоперезапуск после краша ✅ — реальный тест:** `kill -9` процесса → launchd поднял бот
  за ~2 секунды (новый pid). `keepalive`.
- **Переживание перезагрузки Mac ✅:** `runatload` + `keepalive` в plist.
- **Логи в файл ✅:** `logs/bot-stdout.log` + `bot-stderr.log` (launchd) + **новый
  `logs/hook-notify.log`** (мой фикс — теперь провал отправки из хука виден, а не «молча»).
- Утечки/зависания: pty-буфер обрезается до 100 строк (ок). Индикатор «печатает» при долгой
  задаче шлёт `sendChatAction` каждые 4.5с до 3ч — много вызовов, но безвредно (минор).

## 8. Найденные проблемы по приоритету

- 🔴 **Фантомные сессии после рестарта** («Session not found») — **ПОЧИНЕНО**.
- 🔴 **Мусор в списке проектов** (скрытые/подпапки) — **ПОЧИНЕНО**.
- 🟡 **Сессии не переживают рестарт бота** (природа pty) — после фикса состояние честное, но
  после рестарта надо заново `/new`. Полное решение — режим tmux (сессии живут отдельно).
- 🟡 **Сессии привязаны к текущему каталогу** — при `cd` в подпапки во время работы карта
  засорялась (`api`, `dist`). Частично починено (`cwd-changed` больше не пишет сессию); остаток
  от событий completed/waiting в подпапке возможен. Полное решение — привязка к корню проекта.
- 🟡 **`TELEGRAM_WHITELIST` не enforced** — замок только на `CHAT_ID` (безопасно, но переменная зря).
- 🟡 **`alwaysNotify` шлёт «готово» и для десктоп-сессий** (не только телеграм-задач) — размен ради надёжности.
- 🟢 shell-интерполяция имени в `tmux capture-pane` (не в pty-пути).
- 🟢 индикатор «печатает» до 3ч — много API-вызовов.
- 🟢 все правки в `dist` — затрутся при обновлении/переустановке CCGram.
